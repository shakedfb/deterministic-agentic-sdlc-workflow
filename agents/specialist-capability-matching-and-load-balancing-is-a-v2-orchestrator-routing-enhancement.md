# specialist capability matching and load balancing is a v2 orchestrator routing enhancement

V1 orchestrator routing is role-based: each specialist type handles the corresponding task type. The Requirements Analyst Agent receives requirements tasks; the Code Generator Agent receives code generation tasks. This is simple to implement, simple to debug, and sufficient for single-instance specialist teams within the [[optimal multi-agent team size is 3 to 7 specialized agents]] range. The foundational routing function is defined in [[the orchestrator agents role is routing and validation not content generation]] — v2 capability matching extends that routing function rather than replacing it; the orchestrator still does not generate content, it routes with greater precision.

The limitation of role-based routing appears when the system scales. If a code generation task is large enough to benefit from multiple Code Generator Agent instances working on independent sub-tasks, the v1 orchestrator has no mechanism to route across instances — it knows the role but not which instance is available, capable, or appropriate for the specific task. The same limitation applies when specialist agents have differentiated capabilities: if one Code Generator Agent instance has been fine-tuned for frontend tasks and another for backend tasks, role-based routing cannot exploit this differentiation. This limitation is directly observable as the "Unbalanced routing" failure mode in Signal Category 2 of [[what metrics distinguish a well-functioning orchestrator from a coordination bottleneck]]: one specialist saturated while others are idle, because the orchestrator defaults to one agent even when others are capable of the same task — the metric makes the problem measurable, and capability matching is the architectural fix.

V2 routing addresses this with two enhancements. Capability matching routes tasks based on task characteristics matched against specialist capability profiles: does the task require frontend expertise? Route to the frontend-capable specialist. Does it require security-sensitive code? Route to the agent with the strongest security track record. Load balancing adds a second dimension: among equally capable specialists, route to the one with the shortest current task queue.

Both enhancements require infrastructure that does not exist in v1. Capability matching requires a capability registry that maps specialist instances to their strength profiles, updated as operational data accumulates. Load balancing requires real-time state tracking of each specialist's current task load — which in turn requires the [[observability layer with trace-level instrumentation is required before orchestrator metrics become measurable]] to be operational.

The capability registry also has an implementation surface consideration: when routing decisions must evaluate capability profiles and load state at runtime to select among multiple specialist instances, the orchestrator's routing logic grows beyond what role-assignment in CrewAI's hierarchical process can cleanly express. This is one of the scenarios where [[when should LangGraph be chosen over CrewAI for an SDLC agent team]] becomes relevant — LangGraph's conditional edges provide a first-class surface for routing decisions that branch based on accumulated capability and load state, where CrewAI's dynamic delegation is not designed for multi-instance selection logic of this kind.

The deferral to v2 is appropriate because single-instance specialist teams — which cover the entire [[the minimal viable agent set for software-building is requirements, code generation, test generation, and code review]] — do not require load balancing or capability differentiation. The enhancement becomes necessary only when specialists scale to multiple instances, which should not happen until the single-instance baseline is validated. This v2 timing aligns with the co-deferral of [[parallel task execution requires dependency graph analysis and is a v2 concern for SDLC pipelines]] — both enhancements become necessary together when the system scales beyond single-instance sequential processing, and both require the observability layer to be operational before they can be implemented correctly. Neither should be introduced before the [[hybrid sequential-hierarchical orchestration gives predictable flow with dynamic error handling]] baseline is stable, since that architecture is the v1.5 prerequisite whose validation determines whether scaling to v2 is warranted.
